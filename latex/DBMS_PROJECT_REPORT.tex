\begin{abstract}
This report documents \textbf{LearnTrack}, an online learning platform that collects structured educational data---users, instructor profiles, courses, syllabus content, student enrollments, fine-grained learning events (watch time and interaction types), per-item completion progress, and a full quiz engine with attempts and per-answer records. The persistence layer is \textbf{PostgreSQL}, provisioned through \textbf{Supabase}. The schema applies classical relational modelling: lookup tables for enumerations, primary and foreign keys, check constraints, unique business rules (e.g.\ one enrollment per student per course), and soft-delete columns where history must be retained. Heavy analytical queries over multiple fact tables are precomputed using \textbf{five materialized views}, refreshed via a PostgreSQL function callable from the application. \textbf{Triggers} and stored \textbf{functions} (PL/pgSQL) maintain timestamps, instructor-profile consistency, and optional integration with authentication sign-ups.

The application architecture is three-tiered: a \textbf{React (Vite)} client, an \textbf{Express.js} REST API under \texttt{/api/v1}, and the PostgreSQL backend accessed through the Supabase JavaScript client (\texttt{@supabase/supabase-js}). \textbf{Access control} described in this report is \textbf{application-layer}: signed JSON Web Tokens, route-level role checks (\texttt{student}, \texttt{instructor}, \texttt{admin}), and deliberate filtering of queries by \texttt{user\_id}, \texttt{course\_id}, or instructor ownership---not row-level policies inside PostgreSQL.



\end{abstract}

\chapter{Introduction}
\label{ch:intro}

\section{Motivation}
Universities and training providers increasingly rely on platforms that blend content delivery with measurement: which lessons students open, how long they dwell, whether they skip, and how they score on formative quizzes. Collecting such data in \textbf{ad hoc files} leads to inconsistencies, duplication, and reports that drift from operational truth. A \textbf{Database Management System} centralises persistence, exposes a declarative query language (SQL), and enforces \textbf{integrity constraints} so that analytic conclusions rest on coherent facts.

\section{Problem Statement}
The project addressed the problem of designing and implementing a relational database-backed system that simultaneously:
\begin{enumerate}[label=(\alph*),leftmargin=*]
  \item Supports \textbf{daily transactional use}: registration, enrollment, authoring, quiz taking.
  \item Captures \textbf{event-oriented detail} suited to behaviour analysis (activity log with types and durations).
  \item Supports \textbf{aggregated reporting} across students and courses without scanning entire raw histories on each dashboard refresh.
\end{enumerate}

\section{Objectives}
\begin{enumerate}[leftmargin=*]
  \item Produce a \textbf{third-normal-form style} relational model separating entities and lookup/reference tables.
  \item Encode \textbf{business rules} as SQL constraints wherever practical (checks, FKs, uniqueness).
  \item Implement \textbf{read-optimised aggregates} via materialised views keyed to common drill-down dimensions (\texttt{user\_id}, \texttt{course\_id}).
  \item Maintain \textbf{audit-friendly} structures: timestamps, optional soft deletes, granular quiz answers.
  \item Expose data through a documented \textbf{REST API} that mirrors resource boundaries (courses, enrollments, quizzes, analytics).
\end{enumerate}

\section{Scope and Boundaries}
\textbf{In scope:} DDL shape and rationale; DML workflows as exercised by the server; analytic views; triggers and functions bundled with the schema script; deployment configuration at a high level.

\textbf{Out of scope:} Large-scale behavioural science evaluation; predictive models over grades; dedicated mobile-native clients.

\begin{itemize}[leftmargin=*]
  \item \textbf{Integrity}: no dangling enrollments referencing missing users or courses ($\Rightarrow$ FKs).

\end{itemize}

\chapter{Conceptual Database Design}
\label{ch:conceptual}

\section{Core Narrative Model}
\textbf{Teaching graph:}\ A person with instructor capability owns \textit{courses}.\ Each course contains ordered \textit{content} bricks typed as video/document/quiz-marker.

\textbf{Learning graph:}\ A student obtains an \textit{enrollment}, producing a permission to interact with rows tied to that \texttt{course\_id}.

\textbf{Observability graph:}\ Interactions explode into \textit{activity\_log} rows (fine grain) and/or upserts into \textit{content\_progress} (coarse longitudinal).

\textbf{Assessment graph:}\ A course offers \textit{quizzes}; each splits into \textit{questions} \& \textit{options}. A student's \textit{attempt} attaches many \textit{answers}.

\chapter{Logical Design and Normalisation}
\label{ch:logical}

\section{Reduction of Redundancy}
String enumerations lifted to tables with synthetic keys avoids update anomalies: renaming ``active'' status updates one row instead of rewriting every enrollment tuple.

\section{Functional Dependencies (Representative)}
\begin{itemize}[leftmargin=*]
  \item \texttt{user\_id} $\rightarrow$ current profile scalars (\texttt{email} unique enforced separately).
  \item \texttt{course\_id} $\rightarrow$ \texttt{instructor\_id} (determines owner).
  \item (\texttt{user\_id}, \texttt{course\_id}) $\rightarrow$ single enrollment surrogate \texttt{enrollment\_id} (enforced UNIQUE).
\end{itemize}

\section{Decomposition Benefit}
Separate \texttt{quiz\_answers} avoids repeating attempt metadata inside each answer row-only foreign keys \& scores live at attempt granularity.

\section{Referential Actions}
\textbf{Cascade} deletes user-owned sessions/answers when account removed for privacy consistency.\ \textbf{Restrict} on enumerations avoids deleting a referenced status inadvertently.

\chapter{Physical Design}
\label{ch:physical}

\section{Indexing Rationale}
B-tree indexes accelerate:
\begin{itemize}[leftmargin=*]
  \item Instructor dashboard filters: \texttt{courses(instructor\_id)}, \texttt{enrollments(course\_id)}
  \item Student timeline \& analytics windows: \texttt{activity\_log(event\_at DESC)}, \texttt{activity\_log(user\_id)}
  \item Quiz reports: \texttt{quiz\_attempts(user\_id, quiz\_id)}
\end{itemize}

Partial indexing (where \texttt{courses.deleted\_at IS NULL}) shrinks scanned pages when listing catalogue.

\section{Materialized Snapshot Indexes}
Unique composite indexes enable potential \texttt{CONCURRENTLY} refreshes \& fast equality probes on summarized dimensions.

\section{Data Type Justification}
\begin{itemize}[leftmargin=*]
  \item \textbf{TIMESTAMPTZ} prevents ambiguity across client timezones.
  \item \textbf{NUMERIC(5,2)} yields stable decimal aggregates for percentages.
\end{itemize}

\chapter{Relational Schema and Data Dictionary}
\label{ch:dict}

\section{Enumeration / Lookup Tables}
\begin{description}[style=nextline]
  \item[\texttt{activity\_types}]
  Rows: play, pause, skip, seek, complete. PK \texttt{type\_id SERIAL}.
  \item[\texttt{content\_types}]
  video, document, quiz syllabus marker.
  \item[\texttt{enrollment\_statuses}]
  active, completed, dropped.
  \item[\texttt{question\_types}]
  mcq (multi-option), true\_false exactly two labelled options logically.
\end{description}

\section{Core Operational Tables}

\noindent\textbf{users.}
Primary key UUID \texttt{user\_id}. Columns include \texttt{full\_name}, unique \texttt{email}, hashed \texttt{password} mirror (auth may externalise passwords), constrained \texttt{role}, biography and avatar URLs, booleans \& soft-delete \texttt{deleted\_at}, audit timestamps.\ Indexing via PK and UNIQUE email.

\vspace{0.5em}\noindent\textbf{instructors.}
Surrogate PK \texttt{instructor\_id}; UNIQUE FK \texttt{user\_id}$\rightarrow$\texttt{users}. Optional department \& qualification meta.

\vspace{0.5em}\noindent\textbf{courses.}
\texttt{course\_id SERIAL} PK. FK \texttt{instructor\_id}$\rightarrow$\texttt{instructors}. Publication flag, soft-delete, metadata (title\ldots category). Indexed by instructor \& category \& partial published filter.

\vspace{0.5em}\noindent\textbf{enrollments.}
Bridge: (\texttt{user\_id}, \texttt{course\_id}) UNIQUE plus FK pairs. FK \texttt{status\_id} to statuses. Temporal \texttt{enrolled\_at}, \texttt{updated\_at}.

\vspace{0.5em}\noindent\textbf{content.}
Owned by course; FK to \texttt{content\_types}. Ordering \texttt{sort\_order}, optional media URIs \& body text, publish flag.

\vspace{0.5em}\noindent\textbf{activity\_log.}
Fact rows: FK user, FK content (hence implicitly course lineage), FK activity type; non-negative integer \texttt{watch\_time}, \texttt{event\_at}.

\vspace{0.5em}\noindent\textbf{content\_progress.}
Unique (\texttt{user\_id},\texttt{content\_id}); percentage WITH CHECK clamp; last watched timestamp upsert semantics in application logic.

\section{Quiz Relations}
\textbf{quiz} header (title, timers, threshold \texttt{pass\_score}).\ 
\textbf{quiz\_questions} carry type \& points.\ 
\textbf{question\_options} store distributive choices \& boolean correctness.\ 
\textbf{quiz\_attempts} record score/pass timestamp per user \& quiz.\ 
\textbf{quiz\_answers} granularity: (\texttt{attempt\_id},\texttt{question\_id}) unique $\Rightarrow$ prevents double answer rows per prompt.

\section{Session Table}
\textbf{user\_sessions} stores opaque token fingerprints (hashed token), logout \& expiry for optional session revocation analytics.

\section{Materialized Views (Logical Columns)}
\textbf{mv\_performance\_summary} keys (\texttt{user\_id},\texttt{course\_id}); metrics: summed watch seconds, average quiz scores, aggregated completion surrogate.

\textbf{mv\_active\_students} ranks student engagement aggregates.

\textbf{mv\_underperforming\_students} summarises quiz attempt averages \& counts attempts.

\textbf{mv\_skipped\_content} aggregates skip events grouped by content.

\textbf{mv\_completion\_rates} averages progress percent per enrollee per teaching unit.

\section{Summary Table Inventory}
Operational base tables counted: users, lookups (4), instructors, courses, enrollments, content, activity\_log, content\_progress, quiz + 4 quiz detail tables (questions, types options already counted), attempts, answers, sessions = seventeen primary narrative tables excluding views.

\chapter{Database Programming Artefacts}
\label{ch:objs}

\section{Procedural \& Trigger Logic}
\textbf{Role guard trigger} on \texttt{instructors}: inserting profile only if mirrored \texttt{users.role} permissible.

\textbf{Auto-provisioning trigger} inserts instructor row when upgrading role alleviates manual DDL seed mistakes.

\textbf{Timestamp trigger} uniformly bumps \texttt{updated\_at} across mutable domain tables enumerated in DDL loop (\texttt{users}, \texttt{instructors}, \texttt{courses}, \texttt{enrollments}, \texttt{content}, \texttt{quiz}).

\textbf{Auth synch function} inserts app-level user row referencing external identity creation.

\textbf{Bulk refresh routine} sequentially refreshes all analytics materialised relations---invoked remotely by authenticated admin routes.

\section{Illustrative Snippet --- Performance Summary Core Join Pattern}
Listing~\ref{lst:perf} abstracts the pedagogical LEFT join fan-out pattern (identifiers simplified).

\begin{lstlisting}[caption={Representative analytic join motif (conceptual excerpt)},label=lst:perf]
SELECT e.user_id, e.course_id,
       COALESCE(SUM(al.watch_time),0)     AS total_watch_sec,
       COALESCE(AVG(qa.score)::NUMERIC,0)  AS avg_quiz_score,
       COALESCE(AVG(cp.progress_percent)::NUMERIC,0) AS completion_pct
FROM enrollments e
LEFT JOIN content c    ON c.course_id = e.course_id
LEFT JOIN activity_log al
       ON al.content_id = c.content_id AND al.user_id = e.user_id
LEFT JOIN content_progress cp
       ON cp.content_id = c.content_id AND cp.user_id = e.user_id
LEFT JOIN quiz q ON q.course_id = e.course_id
LEFT JOIN quiz_attempts qa
       ON qa.quiz_id = q.quiz_id AND qa.user_id = e.user_id
GROUP BY e.user_id, e.course_id;
\end{lstlisting}

\textbf{Interpretation}: \texttt{LEFT JOIN} ensures an enrolled student-course pair remains even absent optional facts (never attempted quizzes, no logged events).\ Aggregates wrap null extended rows via \texttt{COALESCE}.

\chapter{Transactional Workflows \& API Mapping}
\label{ch:tx}

\section{Enrollment Transaction (Logical Steps)}
\begin{enumerate}[leftmargin=*]
  \item \texttt{SELECT} course ensuring \texttt{is\_published}=true \& not soft-deleted.
  \item \texttt{SELECT} existing enrollment UNIQUE pair quick path conflict.
  \item Resolve active \texttt{status\_id}.
  \item \texttt{INSERT} enrollment tuple.
\end{enumerate}

\section{Quiz Submission}
\begin{enumerate}[leftmargin=*]
  \item Authorisation \& business checks (published quiz, active enrollment unless instructor preview).
  \item Load question catalogue with options \& weighting points map.
  \item Compute fractional score \& pass boolean vs header \texttt{pass\_score}.
  \item \texttt{INSERT} attempt then batched answers.
\end{enumerate}

\section{Endpoint to Table Sketch}
\texttt{POST /courses/:id/content} writes \texttt{content} after resolving FK \texttt{type\_id} lookup.

\texttt{POST /activity} validates content existence then inserts activity row.

\texttt{PUT /progress/:contentId} upserts incremental completion.

\texttt{GET /analytics/active-students} may combine live filtered queries versus materialised view depending query window parameters coded in controllers.

\section{Concurrency Note}
Enrollment uniqueness closed at DB avoids TOCTOU gap between preliminary SELECT and INSERT under parallel requests.

\chapter{Queries, Joins \& Analytics Design}
\label{ch:analytics}

\section{Join Taxonomy Employed}
\begin{description}[style=nextline]
  \item[Inner join chains]
    Example: deriving student roster with statuses requires matching enrollment \& user \& status rows (inner semantics natural).
  \item[Left expansions]
    Performance summarisation \& optional facts (quiz attempts absent still listed).
\end{description}

\section{Aggregation Caveats}
One-to-many joins before grouping can inflate row counts incorrectly if metrics naively \texttt{COUNT(*)} versus distinct entities---controllers sometimes aggregate inside application loops or use relational COUNT DISTINCT equivalents.

\section{Refresh Strategy \& Staleness}
Materialized views are inherently \textbf{stale until refresh}; acceptable if dashboards tolerate minutes-level lag versus raw logs. Operational refresh triggers via RPC align batch cost to low-traffic windows.

\section{Alternate Query Strategy}
Date-scoped analytic endpoints bypass snapshots for correctness of interval filtering---engineering trade-off: freshness vs ingestion performance.

\chapter{Limitations \& Extensions}
\textbf{Stored procedures transactional boundary}: unify multi-step grading as single COMMIT atomic unit.

\textbf{Formal audit}: supplementary append-only log table via triggers capturing old/new JSON diffs.

\textbf{Export adapters}: PostgreSQL COPY or server-side CSV streaming for roster \& attempt matrix.

\textbf{Partitioning horizon}: shard large \texttt{activity\_log} by month if volume crosses operational thresholds.

\chapter{Conclusion}
LearnTrack furnishes an end-to-end illustration of marrying user-facing LMS behaviour with academically rigorous relational database construction: normalised enums, richly constrained bridges, granular facts, summaries materialised for pragmatic analytics retrieval, \& procedural adjuncts safeguarding structural consistency.\ The juxtaposition of transactional APIs and analytic snapshots exemplifies OLTP versus derived reporting patterns common in information systems curricula.\ Future iterations may deepen transactional encapsulation \& external reporting without eroding foundational schema clarity.
